Basic integer types

We encourage the use of following standard C/C++ types, and we state that the following assumptions (no less, no more) on their sizes and limits will be valid for all supported platforms:

Table 1
Name Size(bytes) Min Max Note
char 1 ? ? It can be either signed or unsigned! Use it whereever you dont care of +/- (e.g. in character strings).
signed char 1 -128 127
unsigned char 1 0 255
short, signed short 2 or more -32768 or less 32767 or greater Use "int" if size isn't critical
usigned short 2 or more 0 65535 or greater Use "unsigned int" if size isn't critical
int,
signed int		 4 or more -2147483648 or less 2147483647 or greater
unsigned int 4 or more 0 4294967295 or greater

The use of "long long" type is now prohibited in the portable code.

The "long" type is discouraged to use.



Auxiliary types

Use the "bool" type to represent boolean values. It accepts one of { false, true }.

Use the "size_t" type to represent a size of memory structure, e.g. obtained as a result of sizeof operation.



Floating point types

Use "double" whereever it is possible. Avoid using "float" and dont use "long double" in the portable code.



Fixed-size integer types

Sometimes it is necessary to use an integer type which:

  • has a well-known fixed size(and lower/upper limits)
  • be just the same on all platforms(but maybe a byte/bit order, depending on the processor architecture)
NCBI C++ standard headers provide the following fixed-size integer types:

Table 2
Name Size(bytes) Min Max
Char, Int1 1 kMin_I1 kMax_I1
Uchar, Uint1 1 0 kMax_UI1
Int2 2 kMin_I2 kMax_I2
Uint2 2 0 kMax_UI2
Int4 4 kMin_I4 kMax_I4
Uint4 4 0 kMax_UI4
Int8 8 kMin_I8 kMax_I8
Uint8 8 0 kMax_UI8

Here, "kM*_*" are constants of relevant fixed-size integer type. They are guaranteed to be equal to the appropriate "preprocessor constants" from the old NCBI C headers(*INT*_M*").

The "BigScalar" type

NCBI C++ standard headers also define a special type "BigScalar". The only assumption that can be made(and used in your code) is that "BigScalar" variable has a size which is enough to hold any integral, floating-point or pointer variable (like "Int8", or "double", or "void*). This type can be useful e.g. to hold a callback data of arbitrary fundamental type); but, in general, the use of this type is discouraged.



Encouraged and discouraged types

For the sake of code portability and for better compatibility with the third-party and system libraries, one should follow the following set of rules:

  • Use standard C/C++ integer types "char", "signed char", "unsigned char", "short", "unsigned short", "int", "unsigned int" in any case when the assumptions made for them in Table 1 are enough to fit this case requirements.
  • It is not recommended to use "long" type unless it is absolutely necessary(e.g. in the lower-level or third-party code); and even if you have to -- try to localize the "long" as much as possible.
  • The same(as for "long") is for the fixed-size types enlisted in Table 3. -- If you have to use these in your code, try to keep them inside your modules and avoid mixing them with standard C/C++ types (as in assignments, function arg-by-value passing and in arithmetic expressions) as much as possible.
  • For the policy on other types see in sections "Auxiliary types" and "Floating point types".
